RFC: Event-Driven Notification Delivery

Move from database polling to event-driven delivery for faster notifications and lower database load.

architecture-rfc.md Updated 2026-05-26 ~4 min read

Goals

The notification platform should move from database polling to event-driven delivery so product messages arrive within seconds, not minutes, while reducing read load on the primary database.

Latency <5s

Target p99 notification latency after rollout.

DB load -60%

Expected reduction from removing polling scans.

Rollback 2 wks

Legacy data remains available after cutover.

Proposed Architecture

Application services publish notification events to Kafka. The notification router consumes the topic, chooses the correct channel, and calls the worker through an internal webhook.

Security

The router-to-worker webhook stays inside the private network.

Message Bus Options

Kafka

Best fit because the cluster already exists and replay is required for safe worker restarts.

SQS

Managed and simple, but replay and local testing are weaker for this workflow.

RabbitMQ

Easy to operate for queues, but does not provide the replay model needed here.

Decision

Choose Kafka because the cluster already exists and replay is required for safe worker restarts.

Risks

Duplicate delivery during double-write

Mitigation: require idempotency keys before enabling producer writes.

Likelihood: mediumImpact: high

Consumer lag hides worker failures

Mitigation: alert on lag and pause rollout when lag crosses the threshold.
